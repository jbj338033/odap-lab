# 오답연구소 — 기획 문서

> 개발 착수 전 작성. AI(Claude)와의 대화를 통해 문제 정의 → 솔루션 설계 → 기능 범위를 확정함.

## 1. 문제 정의

### 페르소나

중·고등학생 수험생. 시험을 보고 나면 "틀린 문제를 다시 봐야지" 하지만, 실제로 오답 분석을 체계적으로 하는 학생은 극소수.

### 핵심 고통

| 고통 | 현재 대응 | 한계 |
|------|-----------|------|
| 오답 노트 작성에 30분~1시간 | 손으로 노트에 기록 | 귀찮아서 3회 이상 지속하는 학생이 드묾 |
| 왜 틀렸는지 스스로 판단 불가 | 해설지 확인 | 해설지는 "정답 풀이"만 있고, "내가 왜 틀렸는지"는 안 알려줌 |
| 같은 유형 반복 실수 | 없음 | 약점 패턴을 장기 추적할 수단 자체가 없음 |

### 경쟁 서비스 분석

- **콴다**: 문제 촬영 → 풀이 검색. "왜 틀렸는지"가 아닌 "어떻게 푸는지"에 집중.
- **QuizScanner**: 오답 저장·관리. 저장은 하지만 원인 진단이 없어 복습 효과가 낮음.
- **오답노트 앱들**: 수동 입력 기반. 사진 촬영 자동 추출이 없어 결국 손으로 입력해야 함.

**공백**: "왜 틀렸는지 자동 진단" → "약점 맞춤 연습" → "성장 추적"의 학습 루프가 존재하지 않음.

## 2. 솔루션 가설

> 시험지 사진만 올리면 AI가 오답 원인을 5가지 유형으로 진단하고, 약점에 맞는 연습문제를 자동 생성한다.

이 문제에 AI가 필수인 이유:
- 손글씨 + 채점 표시 + 수식이 혼재된 시험지를 파싱하려면 Vision AI가 유일한 현실적 방법
- "왜 틀렸는지"를 분류하려면 문제 내용, 학생 답, 정답을 모두 이해하고 교육학적 판단을 내려야 함 → rule-based로는 불가능
- 맞춤 연습문제 생성은 학생별 약점 조합이 무한대 → 사전 DB로 커버 불가능

## 3. 핵심 기능 (MVP)

MoSCoW 기준으로 범위 설정:

### Must (반드시 구현)

1. **시험지 사진 → 자동 추출**: Vision AI가 문제·학생답·정답·채점 결과를 구조화
2. **5가지 오답 원인 진단**: 개념 미이해 / 계산 오류 / 부주의 실수 / 지식 부족 / 문제 오독
3. **AI 학습 리포트**: 핵심 약점, 공부법, 잘한 점 요약
4. **맞춤 연습문제**: 틀린 유형에 맞는 4지선다 AI 생성
5. **분석 이력 저장**: 로컬스토리지 기반, 언제든 다시 확인

### Should (있으면 좋음)

6. **대시보드**: 정답률 추이, 오류 유형 분포 시각화
7. **지식 맵**: 개념별 숙련도를 그래프로 시각화

### Could (시간 여유 시)

8. 과목별 필터링
9. 연습문제 난이도 조절
10. PDF 내보내기

### Won't (이번 범위 외)

- 사용자 계정 / 로그인
- 서버 DB 저장
- 다국어 지원

## 4. 사용자 플로우

```
[랜딩 페이지]
    ↓ "시험지 분석하기" 클릭
[업로드 페이지]
    ↓ 시험지 사진 드래그/촬영 (다중 페이지 지원)
    ↓ 제목·과목 입력 (선택)
    ↓ "분석 시작" 클릭
[분석 중] 이미지 압축 → 문제 추출 → 오류 분류 (진행 상태 표시)
    ↓
[분석 결과 페이지]
    ├─ 정답률·정답·오답 통계
    ├─ 오류 유형 파이차트
    ├─ AI 진단 리포트 (스트리밍)
    ├─ 문제별 상세 분석 (접기/펼치기)
    └─ 다음 단계 링크 (연습문제 / 대시보드 / 지식맵)
         ↓
[연습문제 페이지] → AI 생성 → 풀기 → 제출 → 채점·풀이
[대시보드] → 누적 통계, 추이 차트, 약점/강점 태그
[지식 맵] → 개념 노드 그래프, 숙련도 색상 표시
```

## 5. AI 활용 전략

### 개발 도구

| 도구 | 역할 | 선택 이유 |
|------|------|-----------|
| Claude Code (Opus) | 전체 개발 | 코드베이스 전체를 이해하고 아키텍처 일관성 유지 가능 |
| GPT-5.4 (Vision) | 시험지 분석 엔진 | 손글씨·수식·채점표시 인식 + Structured Output 지원 |
| Vercel AI SDK v6 | AI 연동 프레임워크 | generateText + Output.object로 타입 안전한 구조화 출력 |

### AI 파이프라인 (3단계 분리)

왜 한 번에 안 하고 3단계로 나누는가:
- 한 번에 "추출+분류+진단"을 시키면 컨텍스트가 너무 길어져 각 단계의 정확도가 떨어짐
- 단계별 분리하면 실패 시 해당 단계만 재시도 → 토큰 낭비 방지
- 각 단계에 전문 시스템 프롬프트를 설계하여 역할 집중

```
1단계: 추출 (extraction)
   이미지 → Vision API → 문제·답·채점 구조화 데이터
   Zod 스키마로 출력 강제

2단계: 분류 (classification)
   틀린 문제 목록 → 오류 유형 분류 + 원인·해결법
   "모두 같은 유형으로 분류하지 말 것" 제약 조건 포함

3단계: 진단 (diagnosis)
   전체 결과 종합 → 학습 리포트 스트리밍 생성
   간결함 강제 (200자 이내 형식 제한)
```

### 토큰 최적화

- 클라이언트에서 이미지를 1600px 이하로 압축 후 전송 (Vision API 토큰 절감)
- Structured Output으로 자유 텍스트 파싱 실패 → 재시도 완전 제거
- 진단 리포트는 streamText로 점진적 표시하여 체감 대기 시간 단축
